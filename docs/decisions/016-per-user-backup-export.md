# ADR 016: Per-user backup / restore boundaries

## Context

The Family-LAN multi-user plan landed admin-gating for `POST /backup` / `POST /restore`. It did not address what the post-/verify-plan multi-user audit response (`docs/private/MULTI-USER-AUDIT-RESPONSE.md` §6, priority #6, `per_user_backup_export`) explicitly flagged as deferred: backup remains whole-instance, restore is `INSERT ... ON CONFLICT DO NOTHING` additive (silent merges and silent drops), and the existing `GET /api/v1/admin/export` covers only 5 of the ~14 per-user Postgres tables and only one of four Qdrant collections, with no symmetric import.

A user cannot honestly take all their data with them; an operator cannot re-instate one user without restoring the whole instance and risking cross-user contamination. This is the audit's literal B8 finding ("Backup/restore are now admin-gated, but the export/import is still whole-instance. A user can't export just their data, and restore is still additive across rows").

Constraints that shaped the option space: must remain fully local (no cloud); must respect the upcoming `personal/shared/system` scopes (B6); must not silently merge two users with the same `user_id`; must not require a new Docker service; must coexist with the existing whole-instance `/backup` + `/restore` (different unit-of-work, both keep working).

## Decision

Add a per-user export and a per-user import surface alongside the existing whole-instance `/backup` and `/restore`:

- **Export:** `POST /api/v1/me/export` (any authenticated user; admins may pass `target_user_id` in the JSON body to act on behalf — refined from the original `?user_id=X` query param during plan review for consistency with the existing `POST /ingest` admin-on-behalf precedent in `routes/data.py`). Returns an `application/zip` archive containing `manifest.json` (`format_version: 1`, exporting user, `scope_filter: "authored_by_me"` — refined from the original `"personal_only"` once B6's `published_from` projection model confirmed that shared rows preserve the original author's `user_id`, sections+row counts), `postgres/<table>.json` for every per-user Postgres table filtered by `(scope IN ('personal','shared') AND user_id = $me)` for tables with a `scope` column and `user_id = $me` otherwise, `qdrant/<collection>.json` payloads filtered by payload `user_id`, `falkordb/{nodes,edges}.json` for the user's intra-personal subgraph (edges with both endpoints owned by the user; cross-user edges counted in `falkordb_external_edge_count`), and a `users/<user_id>.json` row with `password_hash` and every credential field (anything ending in `_secret`/`_hash`/`_token`/`_credential`/`_jti`) redacted to `null` unconditionally. After every successful export, retention pruning runs in the user's archive directory (keep newest 3; prune over-30-day archives beyond that floor).
- **Import:** `POST /api/v1/admin/user-imports` (admin-only — neutral resource shape, NOT `POST /api/v1/admin/users/{id}/import`, because the operation creates a fresh user with a freshly-minted user_id rather than acting on an existing one). Body: `{archive_path, new_user: {email, password, role}, dry_run: bool}`. Reads a zip from the `_USER_EXPORT_DIR` allowlist; refuses if the target email already exists in `users` (returns 409); refuses if any UUID PK in a parent table (`entities`, `sessions`, `notes`, `audio_memos`, `signals`, `sources`, `deduplication_runs`) collides with an existing destination row (returns 409 with `refusal_reason="uuid_collision_on_parent_table"` — leaf tables fall through to `INSERT ... ON CONFLICT DO NOTHING` with skipped counts in the receipt's `leaf_pk_collisions_per_table`); creates the `users` row via `services/users.py:create_user(...)` with the admin-supplied password (the export never carries credential material; the exported user_id is informational only); then bulk-inserts the per-user Postgres rows wrapped in a single explicit `MetadataStore.transaction()` (added to the port during Pass 0 specifically for this contract), re-embeds Qdrant `documents` from saved payloads (zero-vector upsert for `conversations`/`entities`/`signals` and any document where embedding fails), and re-MERGEs FalkorDB nodes under the freshly-minted user_id. v1 does not restore FalkorDB edges — the receipt carries a `falkordb_edges_not_restored` warning so operators re-derive via re-ingest.
- **Dry-run / preview:** Same endpoint with `dry_run: true`. Reads the archive, validates manifest version + required sections, reports row counts + missing sections + dangling references + would-precondition-pass (including the new `no_parent_pk_collisions` precondition), returns a structured `ImportPlan`, performs zero writes.
- **Audit logging:** Every operation writes structured `audit_log` rows under reserved `__user_export__.*` / `__user_import__.*` action prefixes. Lifecycle covered: export started/completed/failed, dry-run requested/passed/failed, import started/completed/failed/refused. Strict prohibition on logging passwords or any redacted credential field's value, pinned by `_redact_credentials` and explicit-field audit payload construction (no `**user_row` splats).
- **Legacy stability (intentional deviation):** The existing `GET /api/v1/admin/export` (NDJSON, 5 sections) was originally specified to stay byte-for-byte unchanged for one full release with a `DeprecationWarning`. The implementation pivoted to `410 Gone` with a successor pointer per direct user instruction during the implementation pass (recorded in the status history). The 410 response body names `POST /api/v1/me/export` as the successor so pre-existing scripts that hit `/admin/export` get an actionable error instead of a silently-shrinking NDJSON dump. The pre-410 NDJSON dumper is gone for good in this build; the new zip-based export ships only on the new `POST /api/v1/me/export` endpoint. Pinned by `tests/test_user_export_routes.py::test_legacy_admin_export_returns_410_with_successor`.
- **Format is a public contract from `format_version: 1`.** `docs/per-user-export-format.md` documents the schema, the v1-readability commitment, the credentials-redaction guarantee, the two-tier PK collision policy, and the refusal-reason → HTTP status mapping.
- **Whole-instance `/backup` and `/restore` are unchanged** beyond the D11 retroactive CSRF same-origin hardening. They remain the disaster-recovery path AND the household/shared/system snapshot path; the per-user surface is the personal-data-portability path only.
- **Implementation concentration:** All export/import/dry-run/pruning logic lives in one service module (`orchestrator/services/user_export.py`) with small, pure, individually-testable helpers (manifest parse/validate, archive enumeration, pruning decision, dry-run plan builder, zip-slip validator, parent-UUID-collision detector, credential redactor). Route handlers stay thin (~5–10 lines each).
- **CSRF same-origin hardening (D11):** `Depends(require_same_origin)` applied to all new admin POST routes added by this chunk and retroactively to legacy `POST /backup` and `POST /restore`. `POST /api/v1/me/export` also gets the gate. The legacy `GET /admin/export` does not (read-only GET; OWASP guidance). v1 reality: `csrf.require_same_origin` bypasses every Bearer-authenticated caller by design, and v1 `auth_middleware` requires Bearer JWT, so the gate is dormant scaffolding until cookie-auth ships in `cross_device_lumogis_web`.

## Alternatives Considered

- **`POST /api/v1/admin/users/{id}/import` (the original Decision's import shape).** Rejected by user arbitration 2026-04-18 — the path implies importing into an existing user id, which is the exact silent-merge failure mode the chunk exists to prevent. Replaced with the neutral `POST /api/v1/admin/user-imports`.
- **Credential preservation via `?include_credentials=true` opt-in.** Rejected by user arbitration 2026-04-18 — cross-instance JWT_SECRET / refresh-jti state make hash transport effectively useless; opt-in flag adds attack surface for negligible gain. Revisit only if a concrete cross-instance migration use case appears.
- **Round-out only — extend `GET /export` to all per-user tables, ship no import.** Closes the export half but leaves the additive-restore problem the audit explicitly called out.
- **Bundle GDPR-style erasure (`DELETE /api/v1/admin/users/{id}/data`) into the same chunk.** Touch points are 90% shared with import. Doubles the review surface and forces decisions about anonymisation, retention exceptions, and audit-trail implications that are not B8 questions. Better as a follow-up `per_user_account_erasure` chunk.
- **Postgres-native per-user logical export via `COPY ... TO STDOUT WITH (FORMAT csv) WHERE user_id`.** Same architecture, different mechanism. Brings no advantage at family-LAN scale and loses the natural JSON shape for JSONB columns.
- **Docs-only ("your data is yours" policy).** Doesn't honour the audit; the proposed workaround (restore the global archive on a fresh install) is what the audit explicitly flagged as broken in multi-user.
- **Per-tenant Postgres logical replication + per-user replicas.** Hosted-multi-tenant pattern; introduces a new Docker service and per-user databases; ruled out by `family_lan_multi_user` D10.
- **Age-only retention pruning (e.g. delete every archive older than 30 days).** Rejected — a low-frequency user with one annual export would lose their only archive on day 31. Hybrid count-floor (keep newest 3) + age-eligibility (older than 30 days) is the policy.
- **"201 with skip warning" PK collision contract for every UUID collision (the original arbitration-round-1 input).** Refined during arbitration to the two-tier contract — parent tables refuse the entire import (FK-anchored; silent skip would orphan dependent rows), leaf tables keep `ON CONFLICT DO NOTHING` with skip-count warnings (nothing FKs into them, so silent skip is safe).

## Consequences

**Easier:**
- Honest "your data is yours" claim for every authenticated user, not just whole-instance operators.
- Operators can re-instate one user from a portable archive without touching the rest of the stack.
- Multi-user audit B8 closes; the audit-response priority-#6 follow-up ships.
- The same machinery extends naturally to the next chunk (`per_user_account_erasure`) and to any future "device migration" UX.
- Forward-compatible with `personal/shared/system` scopes (B6): manifest already records `scope_filter`.
- `MetadataStore.transaction()` is now available as a port primitive for any future plan needing refuse-and-rollback semantics.

**Harder / impossible:**
- The `format_version: 1` schema becomes a public contract; breaking changes need a v2 with a v1-readable importer or a converter.
- The "import refuses if the destination has a parent UUID collision" precondition forecloses any future "merge two users into one account" use case for archives containing parent-table rows. That would need its own `user_merge` exploration.
- The export is scope-aware (excludes `system` rows; excludes shared rows authored by other users). Forecloses any "household-snapshot" interpretation of the per-user export — household-level snapshots stay in `/backup`.
- Imported users start with stale or zero-vector embeddings until next ingest/turn re-embeds (consistent with existing whole-instance restore behaviour). Documented; not a regression.
- File content (PDFs, audio) is not bundled — the export carries `file_index` paths that must exist on the destination host. Documented as a v1 limitation; tracked as `per_user_file_content_export` if anyone asks.
- FalkorDB edges are not restored on import in v1; operators re-derive by re-ingest. Receipt warning surfaces this so it isn't silent.

**What future chunks must know:**
- `personal_shared_system_memory_scopes` (B6) is live; the `published_from` projection model is load-bearing for the `authored_by_me` scope filter.
- `per_user_file_index_and_ingest_attribution` (B11+B12) is live; the composite `UNIQUE(user_id, file_path)` eliminates the global file_path collision risk on import.
- `per_user_account_erasure` (the natural sibling) should reuse this chunk's manifest schema for the "what was removed" receipt.
- `cross_device_lumogis_web` will eventually surface the "Download my data" button AND ship cookie-auth, which activates the D11 CSRF gates that are dormant in v1; the deferred-and-skipped CSRF blocking tests in this chunk are that team's responsibility to un-skip.

## Revisit conditions

- **`cross_device_lumogis_web` lands cookie-auth** → un-skip the deferred CSRF blocking tests; verify D11 protection becomes active without code changes here.
- **A real `user_merge` use case appears** (e.g. household consolidates two duplicate accounts) → standalone exploration; this chunk's strict refuse-on-parent-UUID-collision stays the default.
- **Per-user data exceeds ~1M rows per table or ~1 GB per export on real households** → revisit `COPY ... TO STDOUT` for the Postgres dump path AND switch the per-table format from "one giant JSON" to "one JSON-line per row" (raise the per-entry size cap or drop it); the architecture stays the same.
- **Lumogis ever pivots to hosted multi-tenant** → revisit per-tenant logical replication; the per-user export contract designed here remains the user-facing surface.
- **A user requests file-content portability** (PDFs/audio bundled in the export) → standalone `per_user_file_content_export`; this chunk's manifest gains a `files/` section and a `bundle_file_content: true` opt-in.
- **GDPR or analogous regulation becomes legally relevant to a Lumogis deployment** → fast-track `per_user_account_erasure` and add the "anonymise vs hard-delete" decision to its own ADR.
