# ADR: Per-user connector permissions (`(user_id, connector)` strict per-user, no sentinel)

**Status:** Finalised
**Created:** 2026-04-20
**Last updated:** 2026-04-22
**Decided by:** composer-2 (Opus 4.7) via /explore per_user_connector_permissions; revised by Claude Opus 4.7 via /review-plan --arbitrate R1; further revised by Claude Opus 4.7 via /review-plan --arbitrate R2; finalised by Claude Opus 4.7 via /verify-plan
**Finalised copy:** `docs/decisions/024-per-user-connector-permissions.md`

## Context

Audit item **A2** (`docs/private/MULTI-USER-AUDIT.md` §1, §3, §11) is the only deferred Phase A item from the family-LAN multi-user audit. The `connector_permissions` table is keyed `UNIQUE(connector)`, and the `routine_do_tracking` table is keyed `UNIQUE(connector, action_type)` with no `user_id` column at all. As a result:

- Any household member who flips `filesystem-mcp` from `ASK` to `DO` flips it for **every** other member from that moment forward.
- Fifteen approvals of `send_email` by Alice silently auto-elevate `routine_do_tracking` such that Bob's next `send_email` is also auto-approved.

The family-LAN multi-user plan (`docs/decisions/012-family-lan-multi-user.md` §"Why connector_permissions stayed global", `family_lan_multi_user.plan.md` Phase 3) deliberately deferred the storage change because:

1. It needed a separate exploration of options (per-user table vs. RLS vs. policy engine vs. overlay table vs. bitmask).
2. The hot-path call contract had to be lifted first (Phase 3 made `permissions.check_permission` and `permissions.routine_check` keyword-only on `user_id` — that groundwork is already in place at every call site; the one remaining gap is `permissions.get_connector_mode`, which silently discards the threaded `user_id` because today's storage doesn't carry it).
3. An admin-gate mitigation (`Depends(require_admin)` on `PUT /permissions/{connector}`) was deemed sufficient for the v1 family-LAN ship — but the audit-response §6 ranks `per_user_connector_permissions` as the **#2 highest-value follow-up chunk** ("the only deferred Phase A item; closes a real household-trust gap").

Constraints shaping the option space:

- **Local-first, single-process Postgres:** application-side filtering is the project convention (locked by ADR 015 / `personal_shared_system_memory_scopes`). RLS for two tables would create an architectural divergence from every other per-user table in the codebase.
- **The `users` table convention** (`010-users-and-roles.sql`): `user_id TEXT NOT NULL DEFAULT 'default'`, no FK to `users(id)` from per-user tables, application-side cascade via `services/users.py::set_disabled`/`delete_user`.
- **The keyword-only `user_id` contract** (`family_lan_multi_user` Phase 3): every audit-shaped call carries `user_id`; `check_permission(*, user_id)` and `routine_check(*, user_id)` already do — `get_connector_mode` is the one remaining storage-shaped gap, and the storage change is the missing piece.
- **The legacy `'default'`-collapse**: every existing `connector_permissions` row carries `user_id = 'default'` (or, post-`db_default_user_remap.py` first run, the bootstrap admin's id). Migrations must answer the data-meaning question, not just the schema question. The chosen answer is **eager backfill at migration time**: cross-join existing global rows × existing real users, so any deployment that previously had `filesystem-mcp:DO` globally keeps `filesystem-mcp:DO` per-user for every then-existing user. No silent regression at upgrade.
- **The `__system__` sentinel from `personal_shared_system_memory_scopes` migration 013** lives on the *visibility* axis (`scope_user_id = '__system__'`). Re-introducing a parallel sentinel on the *user_id* axis (`user_id = '__default__'`) on a different table — even with a different meaning — invites permanent vocabulary confusion that the round-1 addendum to that plan explicitly warned against. **No new sentinel user_id is added by this ADR.**
- **No new Docker services**, no new pip dependencies, no new env vars — this chunk is pure Postgres + Python.

## Decision

Adopt **Option 1** from the exploration: lift both tables to strict per-user storage with no overlay tier and no sentinel user_id.

- **`connector_permissions`** changes from `UNIQUE(connector)` to `UNIQUE(user_id, connector)`. The existing `user_id` column (already there from a half-baked earlier add, defaulting to `'default'`) is the keying column. **Eager backfill** at migration time cross-joins existing global rows × existing real users — gated on **`EXISTS (SELECT 1 FROM users)`** (any user row, including a single enabled user, including disabled users). Sweep policy: ALL legacy `'default'` rows on `connector_permissions` are deleted after fan-out (ASK rows have zero semantic content under the lazy `_DEFAULT_MODE='ASK'` fallback; non-ASK rows have been fanned out). Single-user deployments where `users` is empty (fresh install / `AUTH_ENABLED=false`) skip the fan-out — the `'default'` rows remain for `db_default_user_remap.py` to remap to the bootstrap admin id on first `AUTH_ENABLED=true` boot. **The plan's §"Postgres tables" Phase 3 SQL is the canonical specification** for the gate and sweep policy; this ADR sentence summarises it. [Revised 2026-04-22 — see status history.]
- **`routine_do_tracking`** gains a `user_id TEXT NOT NULL DEFAULT 'default'` column and changes from `UNIQUE(connector, action_type)` to `UNIQUE(user_id, connector, action_type)`. Eager backfill via the analogous cross-join — gated on **`EXISTS (SELECT 1 FROM users)`** AND `auto_approved = TRUE` (per-row predicate; `auto_approved=FALSE` pending counters are NOT fanned out — pre-multi-user `approval_count` belongs to the de-facto single operator, attribution goes to the bootstrap admin via `db_default_user_remap.py`). Sweep policy: ONLY `auto_approved=TRUE` `'default'` rows are deleted after fan-out; `auto_approved=FALSE` `'default'` rows are RETAINED for `db_default_user_remap.py` to remap. `db_default_user_remap.py` learns the new table in its `_SCOPED_TABLES` list so first-time multi-user boots are symmetric with `connector_permissions`. **The plan's §"Postgres tables" Phase 3 SQL is the canonical specification.** [Revised 2026-04-22 — see status history.]
- **`orchestrator/permissions.py`** is rewritten around a strict per-user lookup: `(user_id, connector)` keyed; on miss, fall through to the module-level constant `_DEFAULT_MODE = "ASK"` (unchanged from today). The `_mode_cache` keys move to `(user_id, connector)` and cache both hits and misses (so the lazy-fallback path is hot-cached too). `set_connector_mode`, `routine_check`, `elevate_to_routine`, and `get_connector_mode` all become keyword-only on `user_id`. **`get_connector_mode` lifting is the Phase 3 contract closure.** A new `clear_cache_for_user(user_id)` is called by `users.set_disabled`/`delete_user` for cache hygiene. **Placement of the cache-clear call: AFTER the `MetadataStore.transaction()` block exits successfully** — the cache is process-local Python state, not a DB-side artefact, and cannot be transactionally coupled to Postgres. Mirrors `services.mcp_tokens.revoke()` precedent (also after-commit). [Revised 2026-04-21 — see status history.]
- **Routes** mirror the `mcp_token_user_map` D12 layout — a `me` router gated by `Depends(require_user)` (`GET/PUT/DELETE /api/v1/me/permissions[/...]`) and an admin router gated by `Depends(require_admin)` for admin-on-behalf operations (`GET /api/v1/admin/users/{user_id}/permissions`, `PUT/DELETE /api/v1/admin/users/{user_id}/permissions/{connector}`). The legacy `GET /permissions` in `routes/admin.py` is reframed as an admin-only enumeration over all rows joined to user emails. The legacy `PUT /permissions/{connector}` is **soft-deprecated for one release** (`Deprecation: true` + `Link: rel="successor-version"` headers, mirrors `per_user_backup_export` D6) then `410 Gone` in the next release. The unauthenticated `POST /permissions/{connector}/elevate` in `routes/actions.py` gains `Depends(require_user)` and threads the caller's `user_id` (closes a pre-existing missing-auth gap as a side effect of the Phase 3 wiring).
- **Cascade** on user disable/delete is **retain rows for audit forensics** (mirrors `mcp_token_user_map` D7 exactly). Disable revokes effect immediately by clearing in-memory cache; delete retains rows. Re-enable restores them automatically. Connector preferences carry forensic value — "what mode was Bob's `filesystem-mcp` set to when he was disabled?" is a question audit reviewers actually ask.
- **Backup/restore** adds `routine_do_tracking` to `_USER_EXPORT_TABLES` and `_SERIAL_PK_TABLES`; the exhaustiveness regression test (`tests/test_user_export_tables_exhaustive.py`) is the gate.
- **Telemetry / no-raw-user_id grep gate**: drop `routine_do_tracking` from the SCOPE-EXEMPT exclusion list in `tests/test_no_raw_user_id_filter_outside_admin.py` (it now carries a real per-user `user_id`).
- **Migration** is `<NEXT-FREE>-per-user-connector-permissions.sql`; the integer is **claimed at impl time and the topic index is updated in the same commit**, mirroring the `mcp_token_user_map` D10 / `per_user_connector_credentials` OQ #7 convention exactly. At HEAD on 2026-04-20 the next free integer is `014`, but `014`/`015` are provisionally claimed by `mcp_token_user_map` and `per_user_connector_credentials` respectively; the implementer reconciles at impl time.
- **Mode set in v1** stays `{'ASK','DO'}`; the `mode TEXT NOT NULL CHECK (mode IN ('ASK','DO'))` is preserved with no widening (no `BLOCK` / `SCOPED` / `READ_ONLY` etc.).
- **Cache** stays per-process; multi-process drift is a documented limitation and a deferred follow-up tied to `cross_device_lumogis_web` (orchestrator runs single-worker today).

## Alternatives Considered

Full detail in *(maintainer-local only; not part of the tracked repository)*. One-line rationale per alternative:

- **Option 2 — `(user_id, connector)` per-user rows + `'__default__'` sentinel for household policy overlay** — re-introduces the sentinel-user-id pattern that took dedicated cleanup tooling (`db_default_user_remap.py`) to remove from the schema; collides with the `__system__` vocabulary from migration 013; activates forward-compat scaffolding for a capability not requested by any shipped product feature. If household-wide defaults ever become a real product requirement, do them via `scope='system'` semantics consistent with migration 013, NOT a new sentinel (recorded as deferred follow-up #2 below).
- **Option 3 — separate `connector_permission_overrides` table** — same conceptual costs as Option 2 plus an extra table to keep coherent across migrations, backups, exports, and tests.
- **Option 4 — Postgres Row-Level Security (RLS)** — diverges from the application-side `visible_filter()` convention locked by ADR 015; pgbouncer GUC-leak footgun if Lumogis ever sits behind a pooler.
- **Option 5 — drop Ask/Do entirely and route every write through `review_queue`** — removes a shipped capability documented in the `docs/connect-and-verify.md` user contract.
- **Option 6 — Cedar / Casbin / OPA policy engine** — ~30× the surface area for a 3-bits-per-row policy space; Cedar is currently beta. Reserved as a future migration target if RBAC ever expands beyond admin/user OR a second policy axis appears (e.g. per-resource grants).
- **Option 7 — bitmask permissions in a single `BIGINT`** — 63-bit ceiling is below the (~10 connectors × ~30 action_types) ~300-bit ceiling; the bit-position-to-`(connector, action_type)` registry recreates the indirection it claims to avoid.
- **Option 8 — per-user env vars (`PERMISSIONS_<USER>_<CONNECTOR>=DO`)** — UX-hostile (restart to change), no UI/audit/API.
- **Option 9 — defer further** — leaves Phase A floor incomplete; explicitly contradicts the audit-response §6 ranking.

## Consequences

**Becomes easier:**
- The audit's A2 finding closes — Alice flipping `filesystem-mcp` to `DO` no longer affects Bob.
- The symmetric `routine_do_tracking` gap closes in the same chunk — Alice's 15 approvals stay Alice's.
- The unauthenticated `POST /permissions/{connector}/elevate` hole closes (a Phase 3 audit miss).
- The Phase 3 contract is fully closed at `permissions.get_connector_mode` (the one remaining storage-shaped gap).
- Per-user backup/restore round-trips Ask/Do preferences AND routine-Do approvals automatically (the export already includes `connector_permissions`; this chunk adds `routine_do_tracking`).
- A future "permissions" page in Lumogis Web has a coherent storage substrate to read from.
- The schema becomes the natural future home for per-`(user, connector, action_type)` scopes (a `scopes TEXT[]` column add) without further constraint surgery.
- A future per-user `weekly_review` (audit B7) — when it lands — calls the same `check_permission(*, user_id)` contract and gets per-user enforcement for free.
- Disabled-user audit forensics work — admins can still query "what was Bob's `filesystem-mcp` mode when he was disabled?" via `GET /api/v1/admin/users/{disabled_id}/permissions` (rows retained, mirror `mcp_token_user_map` D7).

**Becomes harder:**
- New users see no per-connector rows by default — the dashboard tile must render `_DEFAULT_MODE = 'ASK'` clearly enough that "no row" doesn't read as "broken". Mitigated by the `web/index.html` user tile spec showing the resolved effective mode with a "(default)" badge when no row exists.
- The legacy `PUT /permissions/{connector}` admin route disappears in N+1 (after the soft-deprecation window) — operators with custom scripts that hit this route will need to migrate to `PUT /api/v1/admin/users/{user_id}/permissions/{connector}`. Documented in `CHANGELOG.md` and `docs/connect-and-verify.md`.
- Multi-worker uvicorn deployments would see in-process cache drift across workers when an admin writes a user's row. Acceptable trade-off for v1 — the orchestrator runs single-worker today (verified against `docker-compose.yml`); revisit if/when multi-worker becomes a deployment shape.
- A future operator who wants household-wide policy defaults will need to either (a) script the admin-on-behalf route to write the same value for every user, or (b) wait for the deferred-follow-up #2 chunk that adds `scope='system'` semantics consistent with migration 013. Option 2 (`__default__` sentinel) was deliberately rejected to avoid sentinel-vocabulary debt; the future direction reuses migration 013's idiom.

**What future chunks must know:**
- The migration number (`<NEXT-FREE>-per-user-connector-permissions.sql`) is **claimed at impl time** in coordination with the in-flight `mcp_token_user_map` (provisional `014`) and `per_user_connector_credentials` (provisional `015`) chunks. The topic index is updated in the same commit.
- **No new sentinel `user_id` value is added by this ADR.** Future per-user table chunks should resist the urge to invent `__default__` / `__shared__` / `__household__` user_id values; the existing `__system__` from migration 013 lives on the *visibility* (`scope_user_id`) axis only. If a future chunk genuinely needs a household-default tier on a per-user table, extend migration 013's `scope` vocabulary rather than introducing a parallel sentinel idiom.
- The keyword-only `user_id` contract on `permissions.get_connector_mode`, `permissions.set_connector_mode`, `permissions.check_permission`, `permissions.routine_check`, and `permissions.elevate_to_routine` is **part of the public contract** of `permissions.py`; future callers must pass `user_id` (the `test_phase3_user_id_contracts.py` pinning suite is extended to cover `get_connector_mode`).
- The cascade-retain pattern matches `mcp_token_user_map` D7 exactly. Future per-user tables that store *preferences-with-forensic-value* (mode toggles, approval thresholds, etc.) should mirror this shape; tables that store *pure preferences without forensic value* (e.g. UI theme) may delete on cascade.
- Future per-`(user, connector, action_type)` scope expansion (deferred follow-up #1) is a non-breaking column add (`ALTER TABLE … ADD COLUMN scopes TEXT[]`); no constraint changes needed.
- The legacy `GET /permissions` survives indefinitely as an admin-only enumeration; the legacy `PUT /permissions/{connector}` survives **one release** with `Deprecation: true` headers (per `per_user_backup_export` D6 pattern), then returns `410 Gone` with successor pointer. `CHANGELOG.md` is the source of truth for the timeline.
- A2 (this ADR) is **independent of** B4 (`per_user_connector_credentials` — the encrypted-credential substrate, separate ADR), B10 (`mcp_token_user_map` — the inverse-direction lookup from MCP token to user, separate ADR), and B6 (`personal_shared_system_memory_scopes` — the shared-memory model, ADR 015). Future explorations referring to "per-user multi-user work" must distinguish these four chunks by audit ID.

## Revisit conditions

Concrete triggers under which this decision should be revisited:

- **A household-wide-default capability becomes a genuine product requirement** (not an A2-mitigation artefact). At that point, extend migration 013's `scope` vocabulary to `connector_permissions` (a `scope='system'` row holds the household default), do NOT introduce a parallel `__default__` sentinel user_id. Open a follow-up exploration if/when this lands.
- **A third role appears** (e.g. `child`, `guest`, `shared`). The strict per-user model needs to gain a per-role default tier ("user override → role default → hardcoded `_DEFAULT_MODE`"). Open a `role_based_connector_defaults` exploration; the schema in this ADR is forward-compatible (one column add).
- **Time-bounded grants are needed** ("Bob can DO `filesystem-mcp` until Friday 6pm"). Adds a `(valid_from, valid_until)` time dimension to the row; not foreclosed but not designed in. Open a `time_bounded_connector_grants` exploration.
- **Cross-user delegation is needed** ("Alice grants Bob temporary DO on `filesystem-mcp`"). Currently no delegation primitive exists; would interact with the shared-memory model from ADR 015.
- **The household exceeds ~20 users** OR **conditional / attribute-based rules become a product requirement** ("DO mode only between 9am and 5pm", "DO mode only when on the home network", per-resource grants like "Bob can write `~/family/` but not `~/`"). At that point, Option 6 (Cedar / Casbin / OPA) starts to earn its complexity; this ADR moves from "current" to "preference layer in front of policy engine".
- **The orchestrator goes multi-worker** (likely tied to a future `cross_device_lumogis_web` follow-up). At that point the per-process `_mode_cache` either needs Postgres `LISTEN/NOTIFY` invalidation or a TTL — the cache contract from this ADR remains correct but its implementation must change.
- **Lumogis adopts a hosted multi-tenant deployment mode.** All four of {A2 (this), B4, B10, B6} get re-evaluated jointly under a hosted-tenant trust model; this ADR likely persists as the tenant-internal preference surface but a tenant-boundary policy engine layer is added in front.
- **A new MCP connector ships with >50 distinct action_types or >10 distinct resource scopes per row** — the hardcoded `('ASK','DO')` binary becomes too coarse; the deferred-follow-up #1 `scopes TEXT[]` column add becomes warranted. Non-breaking.
